품질을 "테스트 개수"가 아니라 실제 장애 예방력으로 본다.
판단 기준은 ./references/quality.md를 로드해서 적용한다.

## 실행 순서

1. **위험 우선 스캔** — 모든 케이스가 아니라 피해가 큰 케이스를 먼저
   - 데이터 손실/중복/오염
   - 권한 우회
   - 침묵 실패 (유저가 모르게 틀린 결과)
   - 결제/인증 오류

2. **배포 전 최소 체크**
   - [ ] happy path
   - [ ] 빈 상태 / 첫 사용 상태
   - [ ] 로딩 / 지연 상태
   - [ ] 실패 / API 오류 / 타임아웃
   - [ ] 권한 없음 / 세션 만료
   - [ ] 기존 데이터 / 예외 입력 / legacy 값
   - [ ] 중복 클릭 / 뒤로가기 / 새로고침

3. **버그 심각도 판단**
   - P1: 서비스 불가 / 데이터 손실 / 결제·권한 오류 → 즉시 수정 또는 롤백
   - P2: 핵심 기능 비정상, 우회 거의 없음 → 당일 대응
   - P3: 기능은 되지만 불편, 제한 조건에서만 → 다음 배포

4. **출력 형식**

```
**피해** [유저가 실제로 잃는 것 / 즉시 vs 누적]
**심각도** P1 / P2 / P3
**재현** [어떤 조건에서 발생하는가]
**검증** [확인한 것 / 아직 확인 안 된 것]
**액션** [즉시 할 것]
```

## 변경 유형별 추가 체크

**UI 변경**

- 텍스트가 길어져도 안 깨지는가
- 모바일 줄바꿈/스크롤/고정폭 문제
- disabled/loading/error 상태 빠지지 않았는가

**API/서버 로직 변경**

- invalid input이 조용히 통과하지 않는가
- 부분 실패 시 상태가 어정쩡하게 남지 않는가

**데이터 변경**

- 기존 레코드가 새 코드에서 안전한가
- 마이그레이션 실패/중단 시 복구 가능한가

**권한/인증 변경**

- UI만 숨기고 서버는 열려 있지 않은가

## 회귀 위험이 높은 신호

- 공용 컴포넌트 수정
- 포맷터/파서/유틸 수정
- 인증/권한/요금제 조건 수정
- 날짜/시간대/locale 로직 수정
- 캐시 키 또는 fetch 조건 수정

## 즉시 지적할 허점

- happy path만 확인하고 배포
- 에러 catch했는데 유저 메시지·로그 모두 빈약
- 기존 데이터/세션/권한 상태 전혀 안 봄
- hotfix 했는데 관측 포인트 없어 재발 감지 불가
- 성공만 확인하고 복구 경로 미확인
- 느린 응답/중복 요청 같은 실제 운영 조건 배제
