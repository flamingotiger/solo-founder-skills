기능 아이디어를 "좋아 보임"이 아니라 실제 유저 문제와 사업 임팩트로 거른다.
판단 기준은 ./references/planning.md를 로드해서 적용한다.

## 실행 순서

1. **문제 재정의** — 요청된 기능이 아니라 유저가 풀고 싶은 문제를 먼저 다시 쓴다

2. **분류**
   - 유저 문제 해결: 실제로 막히는가, 반복적인가
   - 효율 개선: 우회가 가능하지만 마찰이 큰가
   - 착각성 요청: 한 번 들었을 뿐인데 크게 느껴지는가 → 근본 문제 재정의

3. **우선순위 판단**
   - P0: 결제실패/데이터손실/핵심플로우 중단 → 즉시
   - P1: 활성 유저 반복 불편, 전환/리텐션 직접 영향 → 1주 내
   - P2: 우회 가능, 데모에 도움 → 핵심 안정화 후
   - P3: 좋은 인상, 미검증 아이디어 → backlog

4. **출력 형식**

```
**문제** [한 문장으로 재정의]
**분류** P0 / P1 / P2 / P3
**결론** 지금 한다 / 축소한다 / 미룬다
**근거** [유저 영향 / 범위 / ROI 중 핵심 1~2개]
**액션** [다음에 할 것]
```

## 착각성 요청 감지

아래 표현이 나오면 멈추고 근본 문제를 다시 쓴다.

- "있으면 좋을 것 같아서"
- "경쟁사가 있어서"
- "어차피 여기까지 했으니"
- "설정도 같이"
- "관리자 페이지도"
- "나중에 확장할 수 있게 구조만"

## MVP에서 과감히 뺄 것

고급 필터 / 세밀한 커스터마이징 / 권한 레벨 세분화 / 예외 케이스 자동화 / 보기 좋은 대시보드

## 즉시 지적할 허점

- 문제 진술은 있는데 대상 유저가 없음
- MVP인데 설정/권한/관리자 기능까지 붙음
- 효과 측정 방법 없이 기능화 추진
- 핵심 플로우보다 주변 편의 기능이 먼저 올라옴
- 가치 제안이 "더 편리함" "올인원" 수준으로 추상적
