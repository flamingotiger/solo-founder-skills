# Solo Founder Skills

1인 개발자와 소규모 스타트업을 위한 복합 사고 프레임워크 스킬이다.

이 스킬은 여러 역할을 여러 에이전트로 분리해서 토론시키는 방식이 아니라, 하나의 에이전트가 단일 컨텍스트 안에서 높은 안목의 기획, 디자인, 개발, 품질 관점을 동시에 켜고 판단하도록 설계되어 있다.

## 왜 만들었나

큰 조직은 역할이 나뉘어 있기 때문에 각자 전문성을 발휘하기 쉽다. 대신 소통 비용이 늘어나고, 사람 수만큼 조율 비용과 인건비가 커진다.

반대로 1인 사업자나 극초기 스타트업은 혼자서 기획, 디자인, 개발, QA, 우선순위 판단까지 모두 처리해야 한다. 이 구조의 장점은 명확하다.

- 소통 손실이 적다
- 판단 속도가 빠르다
- 인력 비용이 낮다
- 제품의 큰 그림을 한 사람이 일관되게 유지할 수 있다

하지만 단점도 있다.

- 생각이 정리되지 않으면 컨텍스트가 쉽게 엉킨다
- 기능 추가를 할 때 기획, UI, 구현, 회귀 위험을 동시에 점검해야 한다
- 멀티 에이전트처럼 역할을 분리하면 오히려 큰 그림이 끊길 수 있다

이 스킬은 바로 그 문제를 줄이기 위해 만들어졌다.

## 핵심 철학

이 스킬은 "여러 명이 토론하는 회사 방식"을 흉내 내지 않는다.

대신 하나의 에이전트가 아래 관점을 동시에 관리한다.

- Product: 지금 이 일을 해야 하는가
- Planning: 유저 문제와 범위가 맞는가
- Design: UX와 UI 흐름이 자연스러운가
- Engineering: 구현 방식이 기존 코드와 맞는가
- Quality: 회귀 위험과 엣지 케이스는 무엇인가

중요한 점은 역할 분리가 아니라 동시 판단이다.
그리고 그 판단 수준은 각 분야의 평균적인 체크리스트 수준이 아니라 실무자급 기준을 전제로 한다.

- 기획만 따로 보지 않는다
- 개발만 따로 보지 않는다
- 디자인만 따로 보지 않는다
- 품질만 마지막에 붙이지 않는다

하나의 결정에 대해 여러 렌즈를 한 번에 켜고, 서로 충돌하는 지점을 찾아서 더 현실적인 결정을 내린다.
즉, 4개의 전문 렌즈가 서로 견제하고 보완한 뒤, 마지막에 제품 관점에서 지금의 최적해를 선택한다.

## 이 스킬이 해결하려는 문제

예를 들어 혼자 제품을 운영하다 보면 "기능 하나 추가"가 단순한 구현 문제가 아니다.

- 기획적으로 지금 필요한 기능인가?
- 디자인적으로 기존 UX를 망치지 않는가?
- 개발적으로 유지보수 가능한가?
- 품질적으로 회귀 위험은 없는가?

이 스킬은 이런 질문을 분리된 회의로 처리하지 않고, 하나의 실행 흐름 안에서 함께 다룬다.
중요한 것은 단순히 다 본 척하는 것이 아니라, 각 렌즈에서 안목 있는 실무자가 걸러낼 법한 허점을 실제로 걸러내는 것이다.

즉, "혼자서 여러 역할을 한다"를 더 잘하도록 돕는 프레임워크다.

## 언제 사용하나

아래 상황에서 기본 스킬처럼 붙여서 쓰면 된다.

- 새 기능 추가
- 버그 수정
- 구조 변경
- 리팩터링
- 우선순위 판단
- MVP 범위 조정
- 디자인/개발 트레이드오프 결정

특히 다음 같은 요청에 잘 맞는다.

- "이 기능 지금 넣는 게 맞는지 같이 판단해줘"
- "버그는 고치되 큰 그림도 같이 봐줘"
- "기획, 디자인, 개발 관점 다 반영해서 진행해줘"
- "혼자 운영하는 제품 기준으로 ROI까지 봐줘"

## 사용 방식

이 스킬은 복잡한 요청일수록 더 강하게 동작한다.

### 1. 단순 작업

간단한 버그 수정이나 작은 수정은 과도한 분석 없이 바로 실행한다.

예:

```text
이 버튼 클릭 안 되는 버그 고쳐줘
```

이 경우에는 길게 토론하지 않고 빠르게 수정하되, 회귀 위험이나 큰 문제가 있으면 짧게 경고한다.

### 2. 복잡한 작업

새 기능, 설계 결정, 우선순위 판단처럼 큰 작업은 여러 렌즈를 동시에 켜고 판단한다.

예:

```text
온보딩 기능 추가하려는데 기획/디자인/개발/품질 관점 같이 보고 진행해줘
```

이 경우 에이전트는 내부적으로 다음을 함께 본다.

- 지금 해야 하는가
- 범위가 과하지 않은가
- UX 흐름이 자연스러운가
- 기존 코드와 잘 맞는가
- 실패했을 때 어떤 문제가 생기는가

## 출력 방식

복잡한 작업에서는 보통 아래 흐름으로 응답한다.

### 큰 그림

지금 이 작업을 하는 게 맞는지 먼저 판단한다.

### 렌즈 간 긴장

기획, 디자인, 개발, 품질 사이에 충돌이 있을 때만 표면화한다.

예:

```text
개발 공수는 큰데 현재 사용자 임팩트는 낮음
```

### 추천 액션

지금 바로 해야 할 다음 행동을 제안한다.

### 파생 작업

이 결정을 하면 뒤따라오는 후속 작업이 무엇인지 정리한다.

## 멀티 에이전트와의 차이

이 스킬은 멀티 에이전트 협업을 부정하는 것이 아니다. 다만 1인 개발자 관점에서는 다음 문제가 자주 생긴다고 본다.

- 각 에이전트가 자기 역할만 보고 큰 그림을 놓치기 쉽다
- 조율 비용이 늘어난다
- 결론은 많아지는데 실제 실행 판단은 오히려 느려질 수 있다

그래서 이 스킬은 "역할별 토론"보다 "단일 컨텍스트 안의 복합 판단"에 더 초점을 둔다.

정리하면:

- 여러 사람이 일하는 회사를 흉내 내는 방식이 아니다
- 혼자 일하는 창업자의 머릿속 판단 구조를 더 잘 정리하는 방식이다
- 넓게 보고, 바로 실행 가능한 결론까지 연결하는 데 목적이 있다

## 추천 프롬프트 예시

```text

```

```text
[$solo-founder] 이 기능이 지금 필요한지 ROI 기준으로 판단해줘
```

```text
[$solo-founder] 이 버그를 고치되 기획/디자인/개발/품질 관점도 같이 봐줘
```

```text
[$solo-founder] MVP 범위가 커지는 것 같은데 어디서 잘라야 할지 정리해줘
```

## 한 줄 요약

`solo-founder`는 여러 역할을 여러 에이전트로 나누는 스킬이 아니라, 1인 개발자가 하나의 컨텍스트 안에서 큰 그림을 잃지 않고 기획, 디자인, 개발, 품질을 함께 판단하도록 돕는 스킬이다.
