# 개발 렌즈 — 실무형 구현 기준

## 이 문서의 역할
- 코드를 "돌아가게만" 만드는 수준에서 멈추지 않고, 이후 수정 비용과 장애 위험까지 같이 본다.
- 1인 개발자에게 치명적인 숨은 복잡도, 운영 리스크, 데이터 오염 가능성을 먼저 자른다.
- 구조 개선과 빠른 납기를 동시에 만족시키는 최소 해법을 찾는다.

## 구현 전 1차 질문
- 기존 패턴 안에서 풀 수 있는가
- 이미 있는 추상화가 문제를 더 쉽게 푸는가, 오히려 더 꼬이게 하는가
- 새 라이브러리 없이 해결 가능한가
- 이 변경이 상태, 데이터, 권한, 캐시, 라우팅 중 어디를 건드리는가
- 배포 후 문제가 생기면 어디서 관측할 수 있는가

## 먼저 찾을 것

### 유사 구현
- 비슷한 화면, API, 훅, 서비스, 유틸이 이미 있으면 복붙보다 패턴을 따른다.
- 단, 기존 패턴이 이미 실패한 설계면 그대로 복제하지 않는다.

### 데이터 경계
- 입력 검증은 어디서 하는가
- DB 또는 외부 API 응답을 그대로 신뢰하는가
- null, 빈 문자열, legacy 값, enum 추가 시 어떻게 깨지는가

### 상태 경계
- 단일 화면 상태인지
- 여러 컴포넌트/탭/세션 사이에서 공유되는지
- optimistic update나 캐시 동기화가 필요한지

## 구현 원칙

### 1. 흐름을 먼저 단순화
- 분기 수를 줄인다.
- 같은 값이 여러 단계에서 재가공되면 원본 소스를 하나로 모은다.
- UI, 비즈니스 규칙, 데이터 접근을 한 함수에 섞지 않는다.

### 2. 에러는 숨기지 않는다
- `catch` 후 무시 금지
- 실패 시 유저가 볼 메시지와 개발자가 볼 로그를 분리
- 실패가 허용되는지, 재시도가 필요한지, 롤백이 필요한지 결정

### 3. 임시 처리에는 수명 표시
- 임시 예외 분기
- 호환성 fallback
- 데이터 마이그레이션용 코드

위 세 가지는 남길 경우 제거 조건 또는 만료 시점을 적는다.

### 4. 타입과 스키마를 신뢰 경계에 둔다
- 외부 입력, API 응답, 저장 데이터 경계에서 검증
- 내부 로직은 검증된 타입 위에서 단순화
- 타입 시스템이 있어도 런타임 입력 검증이 필요할 수 있다

## 설계 판단 기준

### 지금 추상화해도 되는 경우
- 같은 규칙이 이미 2곳 이상 존재
- 앞으로도 반복이 확실
- 공통화하면 분기보다 이해가 쉬워짐

### 지금 추상화하면 안 되는 경우
- 아직 변형 패턴이 한 개뿐
- "나중에 필요할 것 같아서" 미리 일반화
- 공통화 때문에 호출부가 더 난해해짐

### 리팩터링 우선순위가 높은 경우
- 버그가 같은 계층에서 반복
- 테스트를 쓰기 어려울 정도로 책임이 섞임
- 변경 한 번에 파일 여러 곳이 연쇄 수정됨

## 기술 부채를 보는 기준

### 위험한 부채
- 데이터 정합성에 영향
- 권한 우회 가능성
- 캐시/비동기 경쟁 상태
- 설정 의존도가 높은 하드코딩
- 배포 환경마다 다르게 동작하는 코드

### 감수 가능한 부채
- 명확한 제거 계획이 있는 임시 분기
- 유저 영향이 작은 중복 코드
- 릴리즈를 위한 제한적 하드코딩

감수 가능한 부채라도 아래 중 하나면 다시 본다.
- 다음 작업의 발목을 잡음
- 같은 실수를 반복 유발함
- 새 팀원이 아니어도 미래의 내가 헷갈릴 정도로 숨겨져 있음

## 데이터 변경 시 추가 질문
- 읽기 경로와 쓰기 경로가 모두 안전한가
- 기존 데이터와 신규 데이터가 혼재해도 동작하는가
- 되돌리기 가능한가
- 마이그레이션 없이 배포 가능한가
- 부분 배포 중에도 깨지지 않는가

데이터 구조를 바꾸는데 위 질문에 답이 없으면 구현보다 롤아웃 계획이 먼저다.

## 외부 의존성 추가 기준
- 지금 문제를 30% 이상 단순화하는가
- 유지보수 주체가 명확한가
- 번들, 런타임, 빌드, 라이선스 비용이 감당 가능한가
- 나중에 제거하기 쉬운가

라이브러리 도입 금지 신호:
- 한 함수면 끝날 문제
- 팀이 아니라 나 혼자만 이해하는 복잡한 추상화
- 이미 프로젝트에 유사 도구가 있음

## 구현 후 스스로 리뷰할 것
- 한 함수/컴포넌트가 두 개 이상의 큰 책임을 지는가
- 네이밍만 보면 데이터 흐름이 읽히는가
- 실패 경로가 happy path보다 덜 선명한가
- 로그 없이 장애 원인 추적이 어려운가
- 재사용을 노렸지만 실제로는 호출이 어색한가

## 공수 추정 기준

### 30분 이내
- 지역 수정
- 영향 범위 명확
- 데이터 구조 불변

### 30분~2시간
- 인접한 계층까지 영향
- 테스트 또는 수동 검증 필요
- UI/API 한쪽 이상 동반 수정

### 반나절 이상
- 데이터 모델 변경
- 캐시/비동기/권한 로직 관여
- 롤아웃 순서 고민 필요

반나절 이상이면 "개발이 어렵다"가 아니라 "제품 우선순위를 다시 확인할 시점"으로 본다.

## 완료 기준
- lint/type/test 등 프로젝트 기본 검증 통과
- 핵심 유저 플로우 회귀 없음
- 실패 시 로그 또는 관측 지점이 있음
- 변경 이유와 위험 지점을 한두 문장으로 설명 가능
- 다음 사람이 봐도 왜 이렇게 했는지 추적 가능

## 1인 개발자 엔지니어링 원칙
- 가장 빠른 코드는 보통 가장 짧은 코드가 아니라 가장 적게 퍼지는 코드다.
- 미래 확장성보다 현재 변경 비용과 장애 복구 속도를 우선한다.
- 복잡도를 숨기지 말고 경계에서 끊는다.
- 멋진 구조보다 재발 방지가 우선이다.
