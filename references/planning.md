# 기획 렌즈 — 체크리스트 & 프레임워크

## 우선순위 판단 (빠른 버전)

다음 3가지 질문으로 판단:
1. 이게 없으면 유저가 이탈하는가? → Yes: P1
2. 있으면 유저가 더 쓰는가? → Yes: P2
3. 있으면 좋지만 없어도 되는가? → P3 / Backlog

보조 질문:
- 이 요청은 실제 유저 행동에서 나온 문제인가?
- 한 명의 큰 고객만을 위한 예외인지, 반복되는 패턴인지?
- 직접 구현 대신 문구, 운영, 수동 처리로 당장 완화 가능한가?

## Scope Creep 경고 신호
- "어차피 하는 김에..."
- "이것도 금방 될 것 같은데..."
- "나중에 하면 더 복잡해지니까..."

→ 위 표현 감지 시: 현재 작업과 별도 티켓으로 분리 제안

## 1인 개발자 ROI 기준
- 구현 시간 대비 유저 임팩트
- 1일 이내: 즉시 실행 가능
- 1~3일: 우선순위 확인 필요
- 3일 초과: 반드시 검토 후 결정
- 유지보수 비용이 큰 기능이면 초기 구현 시간이 짧아도 재검토
- 수동 운영을 자동화로 바꾸는 작업은 반복 빈도를 같이 본다

## 범위 결정 원칙
- 지금 스코프: 유저가 요청한 것의 80%만
- 나머지 20%는 v2로
- "좋은 것"보다 "작동하는 것"을 먼저
- 새 기능보다 기존 핵심 플로우 복구가 우선
- 결정을 미루더라도 기준은 남긴다
