# 기획 렌즈 — 실무형 판단 기준

## 이 문서의 역할
- 기능 아이디어를 "좋아 보임"이 아니라 실제 유저 문제와 사업 임팩트로 거른다.
- 1인 개발자가 쉽게 빠지는 scope creep, 과기획, 고객 맞춤 예외 남발을 줄인다.
- 결론을 미루더라도 왜 미뤘는지 설명 가능한 상태를 만든다.

## 높은 수준 판단이란
- 요청된 기능이 아니라 유저가 풀고 싶은 문제를 다시 정의한다.
- "왜 지금 해야 하는가"와 "왜 지금 안 해도 되는가"를 둘 다 설명할 수 있다.
- 기능 추가, 운영 대응, 문구 수정, 실험 중 무엇이 가장 싼 해법인지 비교한다.
- 구현 여부보다 유저 가치, 반복성, 측정 가능성을 함께 본다.

## 낮은 수준 판단 신호
- "필요해 보여서", "있으면 좋아서" 같은 표현으로 우선순위를 정한다.
- 특정 고객 요구를 전체 시장 문제처럼 확대 해석한다.
- 요청받은 UI나 기능 형태를 문제 정의보다 먼저 받아들인다.
- 성공 기준 없이 backlog에 넣거나, 반대로 효과 측정 없이 바로 만든다.

## 먼저 분류

### 1. 유저 문제 해결
- 유저가 현재 플로우에서 실제로 막히는가
- 돈, 시간, 신뢰 중 하나를 잃고 있는가
- 반복적으로 발생하는가

### 2. 효율 개선
- 유저는 어떻게든 하고 있지만 마찰이 큰가
- 내부 운영 비용을 줄이는가
- 자동화하면 반복적으로 시간을 아끼는가

### 3. 착각성 요청
- 한 번 들었을 뿐인데 크게 느껴지는 요청인가
- 만들면 좋아 보이지만 사용 빈도가 낮은가
- 실제 문제보다 솔루션이 먼저 제시된 상태인가

착각성 요청이면 바로 만들지 말고 "근본 문제"를 다시 적는다.

## 우선순위 프레임

### P0
- 결제 실패
- 데이터 손실
- 가입/로그인/주문 등 핵심 플로우 중단
- 법적/보안/신뢰 리스크

처리 원칙:
- 다른 작업을 멈추고 대응
- 임시 우회책과 근본 수정 분리
- 유저 피해 범위와 복구 계획을 먼저 본다

### P1
- 핵심 가치 플로우의 성공률을 분명히 올림
- 활성 유저가 반복적으로 겪는 불편
- 고객 전환이나 리텐션에 직접 영향

처리 원칙:
- 1주 안에 다룰 후보
- "지금 안 만들면 어떤 손실이 생기는가"를 설명할 수 있어야 한다

### P2
- 있으면 더 자주 쓰게 됨
- 불편은 줄지만 우회가 가능함
- 데모/세일즈/인지에는 도움 되나 핵심 문제는 아님

처리 원칙:
- 핵심 플로우 안정화 이후 진행
- 반드시 작은 버전으로 쪼개서 시작

### P3 / Backlog
- 좋은 인상
- 경쟁사 대응용 장식 기능
- 아직 사용 패턴이 검증되지 않은 아이디어

처리 원칙:
- 바로 구현하지 않는다
- 필요하면 실험, 인터뷰, 수동 운영으로 검증

## 문제 정의 템플릿
- 누가 겪는가
- 언제 겪는가
- 무엇이 막히는가
- 지금은 어떻게 우회하는가
- 이 문제를 방치하면 어떤 손실이 누적되는가

위 5개 중 2개 이상 답이 비어 있으면 아직 구현 단계가 아니다.

## 사용해야 하는 이유를 만드는 질문
- 사용자가 지금 쓰는 대안보다 왜 이 서비스를 선택해야 하는가
- 이 서비스가 가장 강하게 해결하는 불편은 무엇인가
- 유저가 첫 사용 30초 안에 체감할 가치는 무엇인가
- 시간이 지나도 남는 이유는 무엇인가: 속도, 정확도, 편의, 신뢰, 비용 절감 중 어디인가
- 경쟁사나 기존 방식과 비교했을 때 한 문장으로 설명 가능한 차이가 있는가

가치 제안은 기능 목록이 아니라 선택 이유여야 한다.

### 약한 가치 제안 신호
- "한 번에 관리 가능"
- "더 편리함"
- "AI 기반"
- "올인원"

위 표현은 보통 추상적이라서, 유저가 왜 옮겨야 하는지 설명하지 못한다.

### 강한 가치 제안의 조건
- 누가 듣는지 명확하다
- 어떤 상황에서 유용한지 드러난다
- 기존 대안보다 무엇이 더 나은지 보인다
- 결과가 추상적 감정보다 구체적 효용으로 설명된다

예:
- 약함: "콘텐츠를 더 쉽게 관리할 수 있습니다."
- 강함: "매일 여러 채널에 같은 내용을 올리는 운영자가 10분 안에 예약, 수정, 재배포까지 끝낼 수 있다."

## 서비스 선택 이유 체크
- 지금 유저가 이 문제를 어떻게 해결하고 있는가
- 그 대안은 왜 불편한가
- 우리 서비스는 그 불편을 얼마나 더 짧고, 빠르고, 안전하게 해결하는가
- 무료 대안 또는 수동 처리와 비교해도 옮겨올 이유가 있는가
- 첫 방문 시 그 이유가 화면과 문구에서 드러나는가

선택 이유가 약하면 기능을 더 추가하기보다 포지셔닝, 첫 경험, 메시지부터 다시 본다.

## 요청이 들어왔을 때 바로 확인할 질문
- 이것은 진짜 유저 요구인가, 특정 고객의 해결 방식인가
- 유저가 원하는 것은 기능인가, 결과인가
- 지금 추가하려는 것이 핵심 플로우를 짧게 만드는가, 길게 만드는가
- 같은 문제를 문구, 기본값, 운영 정책, 온보딩 수정으로 더 싸게 풀 수 있는가
- 만든 뒤 2주 안에 효과를 확인할 수 있는가
- 이 요청이 서비스의 "써야 하는 이유"를 더 선명하게 만드는가, 흐리게 만드는가

## 1인 개발자 ROI 판단

### 높은 ROI
- 1일 안에 만들 수 있음
- 핵심 유저 플로우 성공률이 체감되게 올라감
- 운영 반복 작업을 꾸준히 줄여줌
- 다른 기능의 기반이 되는 정리 작업임

### 낮은 ROI
- 예외 케이스가 많아 유지보수 비용이 큼
- 한 명의 큰 고객 요구인데 일반화가 안 됨
- UI/설정/권한/알림까지 연쇄적으로 범위가 번짐
- 효과 측정이 모호함

ROI가 낮은데 요청 압박만 높은 경우:
- 전면 구현 대신 수동 처리
- 관리자 전용 임시 도구
- 숨겨진 설정으로 제한 배포
- 특정 고객 대상 실험판

## 범위 자르기 규칙

### MVP에서 반드시 남길 것
- 유저가 목표를 달성하는 최소 경로
- 실패 시 복구 또는 우회 가능성
- 운영자가 상태를 확인할 수 있는 최소 관측 수단

### MVP에서 과감히 뺄 것
- 고급 필터
- 세밀한 커스터마이징
- 권한 레벨 세분화
- 예외 케이스 자동화
- 보기 좋은 대시보드

### 자주 생기는 범위 확장 신호
- "어차피 여기까지 했으니"
- "설정도 있으면 좋겠네"
- "나중에 확장할 수 있게 구조만"
- "관리자 페이지도 같이"

이 표현이 나오면 아래로 되묻는다.
- 지금 릴리즈에서 이게 빠지면 유저가 실제로 못 쓰는가
- 나중에 붙여도 구조적으로 손해가 큰가

둘 다 아니면 현재 범위에서 뺀다.

## 요구사항 품질 체크
- 입력과 출력이 명확한가
- 성공 상태와 실패 상태가 정의되어 있는가
- 누가 이 기능을 사용할 수 있는지 정해졌는가
- 빈 상태, 첫 사용 상태, 기존 데이터 상태가 구분되어 있는가
- 완료 후 무엇을 측정할지 정했는가

하나라도 모호하면 "기능 명세 부족"이 아니라 "배포 후 혼선 위험"으로 본다.

## 우선순위 결정을 망치는 패턴
- 시끄러운 고객 한 명의 요구를 전체 문제처럼 해석
- 경쟁사 화면을 보고 이유 없이 따라감
- 사용 빈도보다 구현 재미로 선택
- 근본 문제 대신 요청받은 인터페이스를 그대로 구현
- 성과 측정 없이 기능만 누적

## 좋은 답변의 형태
- 문제를 한 문장으로 다시 정의한다.
- 유저가 이 서비스를 써야 하는 이유를 한 문장으로 요약한다.
- 지금 한다 / 축소한다 / 미룬다 중 하나를 분명히 고른다.
- 결정 이유를 유저 영향, 범위, ROI 기준으로 짧게 댄다.
- 바로 실행 가능한 다음 행동을 남긴다.

예:
- "이 요청의 본질은 필터 추가가 아니라 사용자가 원하는 항목을 빨리 찾지 못하는 문제다."
- "이 서비스의 선택 이유는 정보가 많은 것이 아니라, 사용자가 지금 필요한 항목을 가장 빨리 찾게 해준다는 점이어야 한다."
- "지금은 전체 필터 시스템보다 기본 정렬과 추천 문구 개선이 ROI가 높다."
- "효과를 1주 안에 보려면 검색 성공률이나 클릭률 기준을 먼저 둔다."

## 바로 지적해야 할 허점
- 문제 진술은 있는데 대상 유저가 없음
- 기능은 많은데 왜 이 서비스를 써야 하는지 설명이 안 됨
- 효과를 기대하지만 측정 방법이 없음
- MVP라고 하면서 설정, 권한, 관리자 기능까지 같이 붙음
- 자동화보다 수동 운영이 더 싼데도 무조건 기능화를 추진함
- 핵심 플로우보다 주변 편의 기능이 먼저 올라옴

## 추천 액션 포맷

### 지금 한다
- 핵심 유저 손실이 크고
- 범위가 작고
- 효과를 빠르게 확인할 수 있을 때

### 축소해서 한다
- 문제는 진짜지만 요청된 방식이 과할 때
- 운영/문구/기본값/가이드로 절반 이상 해결 가능할 때

### 지금 하지 않는다
- 반복 패턴이 아직 없음
- 유지보수 비용이 과도함
- 핵심 지표 개선과 직접 연결되지 않음

## 실행 전 마지막 질문
- 이 작업이 끝났을 때 유저 경험에서 무엇이 한 문장으로 달라지는가
- 만들지 않고도 이번 주에 검증할 방법이 있는가
- 내가 이 기능을 한 달 뒤에도 설명하고 유지할 자신이 있는가
