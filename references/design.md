# 디자인 렌즈 — 실무형 리뷰 기준

## 이 문서의 역할
- 화면을 "예쁜가"가 아니라 "이해되는가, 믿을 수 있는가, 행동하게 만드는가"로 평가한다.
- 1인 개발자가 가장 자주 만드는 문제인 어색한 위계, 약한 구조, 상태 누락을 먼저 잡는다.
- 디자인 감각 부족을 감각으로 메우지 않고 반복 가능한 기준으로 보완한다.

## 높은 수준 판단이란
- 스타일보다 정보 구조와 행동 유도를 먼저 본다.
- 화면을 보고 유저가 무엇을 해야 하는지, 무엇을 믿어야 하는지 읽어낸다.
- 빈 상태, 에러 상태, 민감 액션까지 포함해 전체 경험을 본다.
- 예쁜 레퍼런스를 흉내 내는 것이 아니라 제품 맥락에 맞는 밀도와 위계를 선택한다.

## 낮은 수준 판단 신호
- 색, 그림자, 카드 수를 늘리는 방식으로 완성도를 만들려 한다.
- 핵심 행동보다 장식 요소가 먼저 눈에 띈다.
- 기본 상태만 예쁘고 loading, error, empty 상태가 빠져 있다.
- 모바일과 실제 입력 상황을 보지 않고 데스크탑 정적 화면만 본다.

## AI 생성 티가 나는 UI 신호
- radius가 과하게 커서 모든 요소가 둥글고 장난감처럼 보인다
- 텍스트가 많고 문장이 길어서 한 화면에서 읽기 부담이 크다
- 이모지로 위계나 친근함을 대신한다
- 그림자가 많거나 강해서 구조보다 효과가 먼저 보인다
- 의미 없는 그라디언트가 넓게 깔려 정보보다 분위기만 과장된다
- 카드, 배지, 칩, 아이콘이 과도하게 많아 제품보다 데모 시안처럼 보인다

이 신호가 많을수록 "디자인을 했다"기보다 "디자인처럼 보이는 요소를 쌓았다"에 가깝다.
실무 제품 UI는 효과가 많은 화면보다 선택 이유와 구조가 선명한 화면에 가깝다.

## 실무형 시각 디폴트

### radius
- radius는 존재감이 아니라 마감 품질을 위한 요소에 가깝다.
- 기본적으로는 "모서리가 살짝 들어간 정도"가 자연스럽다.
- 버튼, 입력창, 카드 모두 과하게 둥글게 만들기보다 작은 radius를 일관되게 쓰는 편이 제품답다.
- radius가 먼저 보이면 이미 과한 경우가 많다.

실무 감각:
- sharp와 pill의 중간이 아니라, sharp에서 약간만 완화된 쪽에 둔다.
- 둥근 인상이 브랜드 핵심이 아닌 이상 모든 요소를 같은 큰 radius로 통일하지 않는다.

### shadow
- shadow는 구조를 띄우는 보조 장치이지 존재감을 뽐내는 효과가 아니다.
- 강한 blur, 큰 spread, 다층 그림자는 대개 데모 느낌을 만든다.
- 카드 구분은 shadow보다 간격, 경계선, 배경 톤 차이로 먼저 해결한다.

### gradient
- gradient는 브랜딩이나 특정 강조 구간에서만 제한적으로 쓴다.
- 화면 전체 배경이나 카드 다수에 반복 적용하면 실무 제품보다 콘셉트 시안처럼 보인다.
- gradient 없이도 성립하는 구조가 먼저다.

### text
- 설명 문장을 계속 추가하기보다 제목, 보조문구, CTA만으로 이해되게 줄인다.
- 한 화면이 친절해 보이기 시작하면 오히려 읽기 비용이 올라가는지 의심한다.
- 핵심 행동 주변 문장은 짧고 단정하게 유지한다.

### icon and emoji
- 이모지는 정보 구조나 브랜드 톤을 대신하면 안 된다.
- 아이콘도 장식보다 의미 전달이 우선이다.
- 텍스트와 레이아웃만으로도 화면 목적이 보여야 한다.

## 디자인의 우선순위

### 1. 이해
- 첫 3초 안에 이 화면의 목적이 읽히는가
- 유저가 지금 해야 할 행동이 하나로 보이는가
- 중요한 정보와 보조 정보가 섞이지 않는가

### 2. 신뢰
- 간격, 정렬, 상태 표현이 안정적으로 보여 제품 완성도가 느껴지는가
- 에러, 로딩, 빈 상태가 준비되어 있어 불안하지 않은가
- 금융, 결제, 계정, 삭제 같은 민감 행동에서 충분히 확인을 주는가

### 3. 속도
- 읽는 순서가 자연스러운가
- 폼 입력, 선택, 확인이 짧은 단계로 이어지는가
- 불필요한 카드, 모달, 설명 문장 때문에 속도가 느려지지 않는가

## 빠른 자기진단
- 핵심 CTA가 시선 흐름 안에서 가장 먼저 잡히는가
- 제목, 본문, 메타 정보의 단계가 타이포와 간격만으로도 구분되는가
- 강조색은 한 화면에서 진짜 중요한 것에만 쓰였는가
- 카드/배경 효과를 다 빼도 정보 구조가 유지되는가
- 라벨 없이도 아이콘 의미가 충분히 명확한가
- radius, shadow, gradient를 줄였을 때 오히려 더 제품답게 보이지 않는가

## 화면 구조 리뷰

### 정보 계층
- 가장 중요한 것 1개
- 그 다음 이해를 돕는 것 2~3개
- 마지막에 보조 정보

이 순서가 시각적으로도 그대로 반영되어야 한다.

### 레이아웃
- 시작선이 정렬되어 있는가
- 같은 수준의 블록은 같은 폭과 같은 여백을 가지는가
- 섹션 간격과 내부 간격이 다른 의미를 가지는가

좋은 화면은 간격만 봐도 구조가 읽힌다.

### 밀도
- 정보가 많은데 구분이 약하면 답답하다
- 정보가 적은데 여백만 넓으면 비어 보인다
- 밀도는 "많고 적음"이 아니라 "묶음과 분리의 정확도" 문제다

## 컴포넌트 사용 기준
- 새 UI보다 기존 컴포넌트 재사용 우선
- 재사용이 안 되면 해당 컴포넌트가 너무 특수한지, 새 요구가 진짜 공통 패턴인지 판단
- 버튼, 입력창, 배지, 모달, 토스트의 변형이 과도하게 늘어나지 않게 한다
- 시각 효과 차이보다 구조 차이로 컴포넌트 의미를 구분한다

새 변형을 만들 때 질문:
- 다른 화면에서도 다시 쓸 가능성이 있는가
- variant 추가가 기존 의미 체계를 해치지 않는가
- 이름만 추가되고 실제 차이는 미미하지 않은가

## 실무형 텍스트 원칙
- 버튼은 "확인"보다 행동을 말한다
- 제목은 화면 목적을 말하고, 설명은 다음 행동의 불안을 줄인다
- 에러 문구는 원인보다 유저가 할 수 있는 조치를 우선 보여준다
- 빈 상태는 "왜 비었는지"와 "다음에 뭘 하면 되는지"를 같이 말한다
- 한 화면에서 설명이 길어지면 레이아웃 문제가 아닌지 먼저 의심한다
- 텍스트가 많을수록 친절한 것이 아니라 핵심 가치가 흐려질 가능성이 높다
- 섹션마다 제목, 설명, 부연 설명이 모두 붙는 구조를 경계한다
- 문단이 길어지면 문장을 다듬기 전에 정보 자체를 덜어낼 수 있는지 본다

## 실무형 스타일 절제 규칙
- 기본값은 "조금 덜"이다.
- radius는 작게, shadow는 얕게, gradient는 드물게, 색상 수는 적게 간다.
- 강조는 한 화면에서 한두 군데면 충분하다.
- 시각 효과를 빼도 구조가 유지되지 않으면 효과를 더하는 대신 구조를 다시 잡는다.
- 화면이 어색할 때는 새 장식을 추가하지 말고 텍스트, 간격, 정렬, 폭부터 줄인다.

## 상태 설계

### 반드시 있는 상태
- default
- hover / focus
- active / selected
- disabled
- loading
- success
- error
- empty

상태가 빠지면 평소엔 괜찮아 보여도 실제 사용에서 품질이 급락한다.

### 민감 액션 추가 기준
- 삭제, 결제, 탈퇴, 공개 전환은 시각적으로 일반 액션과 구분
- irreversible action이면 결과를 미리 설명
- destructive action 버튼은 주변 액션보다 더 신중하게 배치

## 폼 UX 기준
- 한 줄에 여러 결정을 몰아넣지 않는다
- placeholder는 예시이지 라벨 대체가 아니다
- 필수/선택 구분이 명확해야 한다
- 에러는 제출 후 한꺼번에 터뜨리기보다 가능한 빨리 알려준다
- 입력 성공/실패 후 다음 단계가 보여야 한다

## 반응형 기준

### 모바일
- 핵심 액션이 첫 화면 안에 있거나 자연스럽게 도달 가능해야 한다
- 터치 타겟 최소 44px
- 2열 레이아웃은 이유가 없으면 1열로 단순화
- 길어진 텍스트와 숫자, 태그, 버튼 wrap을 반드시 확인

### 데스크탑
- 넓어진 공간을 단순히 늘리지 말고 정보 그룹을 분명히 나눈다
- 최대 폭을 관리하지 않으면 읽기 밀도가 무너진다
- 사이드 영역이 있다면 본문보다 중요해 보이지 않게 한다

## 접근성 기준
- 색만으로 상태를 구분하지 않는다
- focus ring은 숨기지 않는다
- 대비가 약한 보조 텍스트 남발 금지
- 키보드로 이동 가능한가
- 스크린리더가 읽었을 때 순서가 논리적인가

접근성은 별도 옵션이 아니라 기본 사용성이다.

## 디자인 감각이 부족할 때 먼저 볼 것
- 간격 규칙이 있는가
- 정렬 기준선이 살아 있는가
- 위계가 두세 단계로 명확한가
- 색이 정보 구조를 돕는가, 방해하는가
- 상태가 빠진 컴포넌트가 없는가

## 금지 패턴
- 허전해서 장식 요소를 추가
- 복잡한 문제를 색, 그림자, 그라데이션으로 해결하려 함
- 모든 텍스트가 비슷하게 보여 읽는 순서가 없음
- 카드 안에 카드, 섹션 안에 섹션을 과도하게 중첩
- 모바일 문제를 데스크탑 기준 시안으로 덮음
- 과한 radius로 제품 성격과 무관하게 모든 요소를 둥글게 만듦
- 이모지, 배지, 강조 문구로 위계 부족을 가림
- 그림자와 블러, 그라디언트로 구조 빈약함을 덮음
- 설명 문단을 계속 추가해 인터페이스 대신 문서처럼 만듦
- radius, shadow, gradient를 동시에 강하게 써서 화면 밀도를 흐림
- 브랜드 기준 없이 귀엽거나 둥근 인상을 기본값처럼 사용함

## UI 리뷰 시 우선 지적할 것
- 첫 화면에서 핵심 행동이 약한 부분
- 구조보다 스타일이 먼저 튀는 부분
- 타이포 단계와 간격 단계가 뒤섞여 보이는 부분
- 상태가 없어 실제 사용 순간에 불안정해지는 부분
- 모바일에서 줄바꿈, 고정폭, 터치 영역 때문에 무너지는 부분

## 좋은 답변의 형태
- 예쁘다/별로다 대신 무엇이 이해를 방해하는지 말한다.
- 위계, 간격, 정렬, 상태, 밀도 중 어디가 문제인지 구체화한다.
- 핵심 행동을 더 선명하게 만드는 수정 방향을 준다.
- 기존 패턴을 살릴지 새 패턴이 필요한지 근거를 댄다.
- 필요하면 "무엇을 제거해야 하는지"까지 분명히 말한다.

예:
- "지금 화면은 CTA보다 보조 카드가 더 강하게 보여 읽는 순서가 뒤집혀 있다."
- "문제는 컬러가 아니라 섹션 간격과 타이포 단계가 같아서 정보 묶음이 안 보이는 점이다."
- "이 경우 새 컴포넌트를 만들기보다 기존 폼 패턴에 empty/error 상태만 보강하는 편이 안전하다."
- "radius와 shadow가 과해서 제품보다 랜딩 시안처럼 보여, 모서리와 효과를 먼저 줄이는 편이 맞다."

## 바로 지적해야 할 허점
- 제목과 본문, 보조 정보가 거의 같은 무게로 보임
- 버튼 라벨이 행동을 설명하지 못함
- 빈 상태가 설명만 있고 다음 행동이 없음
- destructive action이 일반 액션과 같은 톤으로 배치됨
- 반응형에서 줄바꿈, 터치 타겟, 최대 폭 관리가 무너짐
- radius, 그림자, 그라디언트는 강한데 정작 구조와 위계는 약함
- 텍스트를 줄이지 못해 핵심 CTA보다 설명이 더 큰 비중을 차지함
- 이모지나 장식 아이콘 없이는 화면의 성격이 설명되지 않음
- 모든 컴포넌트가 같은 큰 radius를 가져 제품 톤이 가벼워 보임
- 카드 경계가 shadow에만 의존해 레이아웃이 흐릿해 보임

## 레벨업 훈련 루프
- 잘 만든 제품 UI 하나를 보고 "왜 안정적으로 읽히는지"를 구조 기준으로 5줄 적는다.
- 같은 화면을 코드나 Figma로 재현하고 차이를 간격, 위계, 정렬 기준으로 비교한다.
- 내가 만든 화면에서 가장 어색한 3곳을 찾고, 각각이 구조 문제인지 스타일 문제인지 구분한다.
- 참고 화면은 복제 대상이 아니라 판단 기준을 배우는 재료로 쓴다.
