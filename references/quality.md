# 품질 렌즈 — 실무형 검증 기준

## 이 문서의 역할
- 품질을 "테스트 개수"가 아니라 실제 장애 예방력으로 본다.
- 1인 개발자가 놓치기 쉬운 회귀, 데이터 오염, 관측 부재를 릴리즈 전에 걸러낸다.
- 자동화가 부족해도 위험 기반 검증으로 배포 자신감을 만든다.

## 높은 수준 판단이란
- 버그 존재 여부보다 유저 피해 규모와 탐지 지연을 먼저 본다.
- 테스트가 없더라도 어떤 리스크를 어떤 방식으로 검증했는지 남긴다.
- 재현, 영향 범위, 복구 가능성을 함께 본다.
- 증상을 덮는 수정과 재발을 줄이는 수정을 구분한다.

## 낮은 수준 판단 신호
- happy path만 확인하고 배포한다.
- "로컬에서 됨"을 품질 근거처럼 쓴다.
- 버그를 고치고도 왜 깨졌는지, 어디까지 영향이 있는지 모른다.
- 자동화 테스트가 없다는 이유로 검증 자체를 생략한다.

## 품질을 보는 순서

### 1. 유저 피해
- 실패하면 유저가 무엇을 잃는가
- 즉시 눈치채는가, 나중에 누적되는가
- 우회 가능한가

### 2. 재현성
- 항상 재현되는가
- 특정 계정/브라우저/데이터에서만 나는가
- timing issue 또는 race condition 가능성이 있는가

### 3. 탐지성
- 장애가 나면 내가 바로 알 수 있는가
- 유저 제보 전에는 모를 가능성이 큰가
- 로그, 알림, 관리자 화면 등 관측 수단이 있는가

품질은 "버그 개수"보다 "피해 규모 x 탐지 지연"으로 본다.

## 모든 변경에 적용하는 최소 체크
- [ ] happy path
- [ ] 빈 상태 / 첫 사용 상태
- [ ] 로딩 상태 / 지연 상태
- [ ] 실패 상태 / API 오류 / 타임아웃
- [ ] 권한 없음 / 세션 만료
- [ ] 기존 데이터 / 예외 입력 / legacy 값
- [ ] 중복 클릭 / 중복 요청 / 뒤로가기 / 새로고침

## 변경 유형별 추가 체크

### UI 변경
- 텍스트 길이가 길어져도 안 깨지는가
- 모바일 줄바꿈, 스크롤, 고정폭 문제는 없는가
- disabled, loading, error 상태가 빠지지 않았는가

### API/서버 로직 변경
- invalid input이 조용히 통과하지 않는가
- idempotency가 필요한 요청인가
- 부분 실패 시 상태가 어정쩡하게 남지 않는가

### 데이터 변경
- 기존 레코드가 새 코드에서 안전한가
- 새 레코드가 구버전 코드와 함께 있어도 견디는가
- 마이그레이션 실패나 중단 시 복구 가능한가

### 권한/인증 변경
- UI만 숨기고 서버는 열려 있지 않은가
- role이 없는 계정, 오래된 토큰, 탈퇴 직전 상태를 고려했는가

## 회귀 분석 방법

### 영향 범위 탐색
```bash
rg -n "변경된심볼명" .
```

함수 호출만 보지 말고 아래도 함께 본다.
- import chain
- route 연결
- feature flag
- env/config 분기
- background job / webhook / cron 연동

### 회귀 위험이 높은 신호
- 공용 컴포넌트 수정
- 포맷터, 파서, 유틸 수정
- 인증/권한/요금제 조건 수정
- 날짜, 시간대, 통화, locale 로직 수정
- 캐시 키 또는 fetch 조건 수정

## 버그 심각도 판단

### P1
- 서비스 불가
- 데이터 손실/중복/오염
- 결제/권한/보안 오류
- 고객 신뢰에 즉시 타격

처리:
- 즉시 수정 또는 롤백
- 원인 파악 전이라도 피해 확산 차단

### P2
- 핵심 기능 비정상
- 우회가 거의 없음
- 일부 계정/환경에 국한돼도 business critical

처리:
- 당일 대응
- 재현 조건, 영향 범위, 임시 우회책 확보

### P3
- 기능은 되지만 불편
- UI 상태 오류
- 제한된 조건에서만 깨짐

처리:
- 다음 배포 후보
- 반복되면 자동화 테스트 또는 guard 추가

## 테스트 전략

### 자동화 테스트를 우선할 곳
- 결제, 생성, 삭제, 권한, 계산식
- 과거에 두 번 이상 깨진 영역
- 데이터를 잘못 쓰면 복구가 어려운 영역

### 수동 검증으로도 충분한 곳
- 단순 문구 변경
- 지역적인 스타일 수정
- 재사용 범위가 낮은 화면

### 테스트를 쓰지 않아도 남겨야 할 것
- 재현 절차
- 기대 결과
- 실제 확인 결과

이 세 개가 없으면 다음 장애 때 같은 조사 비용을 다시 낸다.

## 배포 전 확인 질문
- 내가 직접 핵심 플로우를 끝까지 수행해봤는가
- 실패 시 로그에서 바로 흔적을 찾을 수 있는가
- feature flag, env, seed 데이터 차이로 운영에서만 깨질 가능성은 없는가
- 롤백 또는 hotfix가 가능한가

## 디버깅 시 품질 관점
- 이 버그는 코드 버그인가, 데이터 버그인가, UX 설계 버그인가
- 수정 후 같은 계열 문제가 다른 입력에서도 터지지 않는가
- 증상만 가렸는가, 재발 조건을 없앴는가

## 좋은 답변의 형태
- 유저 피해를 먼저 말한다.
- 어떤 조건에서 재현되는지 명확히 적는다.
- 이번 수정이 막는 범위와 아직 남아 있는 위험을 구분한다.
- 자동화 테스트, 수동 검증, 관측 보강 중 무엇을 했는지 남긴다.

예:
- "이번 버그는 일부 화면 깨짐이 아니라 중복 제출로 데이터가 두 번 생성되는 문제라 P1에 가깝다."
- "수정으로 신규 입력은 막았지만 기존 오염 데이터는 읽기 경로에서 추가 방어가 필요하다."
- "테스트는 없지만 재현 절차 3단계를 고정해 회귀 확인을 마쳤다."

## 바로 지적해야 할 허점
- 실패 조건은 있는데 영향 범위가 없음
- 테스트 통과를 말하지만 어떤 핵심 플로우를 검증했는지 없음
- 기존 데이터/세션/권한 상태를 전혀 보지 않음
- hotfix는 했지만 관측 포인트가 없어 재발 감지가 안 됨
- 에러 상태 UI가 없어 유저는 침묵 실패만 경험함

## 1인 개발자 품질 원칙
- 모든 것을 자동화하려 하지 말고, 치명적인 것부터 자동화한다.
- 테스트가 없으면 관측이라도 남긴다.
- 릴리즈 자신감은 감이 아니라 체크된 리스크 개수에서 나온다.
- 버그를 고칠 때는 "이번 한 번"이 아니라 "다시 같은 비용을 낼 것인가"를 기준으로 본다.
