코드를 "돌아가게만" 만드는 수준에서 멈추지 않는다.
변경 비용, 운영 리스크, 데이터 오염 가능성을 함께 본다.
판단 기준은 ./references/engineering.md를 로드해서 적용한다.

## 실행 순서

1. **요청 분류**
   - 단순 구현: 기존 패턴 확인 후 바로 작성
   - 버그 수정: 디버깅 프로세스 실행
   - 설계 결정: 경계 분석 후 판단
   - 리팩터링: 위험 범위 확인 후 진행

2. **구현 전 체크 (설계/리팩터링 시)**
   - 기존 패턴 안에서 풀 수 있는가
   - 상태/데이터/권한/캐시/라우팅 중 어디를 건드리는가
   - 배포 후 어디서 관측할 수 있는가

3. **디버깅 프로세스**
   - 분리: UI표시 / 서버계산 / 저장데이터 / 입력시점 중 어디인가
   - 관측: 두 번 실패하면 추측 중단 → 로그/쿼리 먼저
   - 분리 처리: 신규 오염 차단 ≠ 기존 오염 방어

4. **출력 형식**

```
**경계** [어디를 건드리는가: 상태/데이터/권한/캐시]
**위험** [변경 비용 / 회귀 위험 / 롤백 가능 여부]
**구현** [코드 또는 방향]
**관측** [배포 후 어디를 볼 것인가]
```

## 스택 특화 규칙 (Next.js + Supabase + Vercel)

### Supabase

- `@supabase/ssr` 사용 필수. `@supabase/auth-helpers-nextjs` 사용 금지 (deprecated)
- 쿠키: `getAll` / `setAll`만 사용. `get` / `set` / `remove` 단독 사용 금지
- RLS는 서버 컴포넌트에서 검증. 클라이언트 단독 신뢰 금지
- Vercel 동적 IP 대응: Atlas/Supabase IP 허용 범위 `0.0.0.0/0` 필요

### Next.js App Router

- `use client` 최소화. Server Component 우선
- 데이터 패칭은 서버 컴포넌트에서. `useEffect` 데이터 패칭 금지
- `error.tsx` / `loading.tsx` 항상 준비
- 환경변수: `NEXT_PUBLIC_` 접두사는 클라이언트 노출됨 주의

### LemonSqueezy (결제)

- Webhook 수신 시 서명 검증 필수
- 결제 상태는 DB에 저장 후 참조. 클라이언트 파라미터 신뢰 금지

## 즉시 지적할 허점

- 에러를 catch하고 로그/유저 메시지 없이 무시
- DB/API 입력을 런타임 검증 없이 신뢰
- 데이터 모델 변경인데 롤아웃 순서/호환성 고려 없음
- 공용 컴포넌트 수정인데 회귀 범위 확인 없음
- 쓰기 경로 방치한 채 읽기 경로만 보정
- `use client` 남발로 RSC 이점 포기
