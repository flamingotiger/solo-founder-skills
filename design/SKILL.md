UI를 "예쁜가"가 아니라 "이해되는가, 믿을 수 있는가, 행동하게 만드는가"로 판단한다.
판단 기준은 ./references/design.md를 로드해서 적용한다.

## 실행 순서

1. **분류** — 요청이 생성인지 리뷰인지 먼저 구분한다
   - 생성 요청: 기준을 적용해 처음부터 올바르게 만든다
   - 리뷰 요청: 아래 리뷰 프로세스를 실행한다

2. **리뷰 프로세스** (화면/코드가 주어진 경우)
   - 1차: AI 생성 티 신호 스캔 → 발견 즉시 지적
   - 2차: 우선순위 순서로 평가 (이해 → 신뢰 → 속도)
   - 3차: 필수 상태 체크 (default/hover/focus/active/disabled/loading/success/error/empty)
   - 4차: 수정 방향 제시 — "더 예쁘게"가 아니라 "더 시스템 있게"

3. **레퍼런스 활용** (새 화면 / 큰 UI 변경 / 리디자인일 때)
   - 프로젝트와 톤이 맞는 레퍼런스 2~3개를 먼저 찾는다
   - 레퍼런스는 그대로 복제하지 말고 정보 구조, 시각적 위계, 인터랙션 패턴만 추출한다
   - 현재 제품의 패턴, 브랜드 톤, 기술 제약과 충돌하면 레퍼런스보다 제품 일관성을 우선한다

4. **출력 형식**

```
**이해** [3초 안에 목적이 보이는가 / CTA가 시선 흐름 안에 있는가]
**신뢰** [상태 표현 안정성 / 빠진 상태]
**속도** [읽는 순서 / 불필요한 요소]
**지적** [즉시 고쳐야 할 것 1~3개]
**방향** [수정 우선순위]
```

## 생성 시 적용 기준

코드나 컴포넌트를 만들 때 ./references/design.md의 시각 디폴트를 적용한다.
새 화면이나 큰 UI 변경이면 위 레퍼런스 활용 규칙을 먼저 적용한 뒤 시각 디폴트를 입힌다.
다만 이 기준은 "무조건 금지"가 아니라 기본값이다. 브랜드, 기존 제품 톤, 정보 구조상 더 나은 이유가 있으면 예외를 허용한다.

- radius: 작게. 모서리만 살짝 둥글게.
- shadow: 구조 보조 수준에서만 얕게. 강한 shadow나 다층 shadow는 피한다.
- gradient: 브랜딩 구간이나 강한 이유가 있을 때만 제한적으로 사용한다.
- color: neutral + key color 1개를 기본으로 하되, 기존 브랜드 시스템이 있으면 그쪽을 우선한다.
- card: 독립된 묶음에만 우선 사용한다.
- text: 설명 문단 최소화

## 즉시 지적할 신호

발견하면 리뷰 첫 줄에 꺼낸다.

- radius 과대 → 장난감처럼 보임
- 이모지로 위계 대체
- 강한 shadow/gradient → 구조보다 효과가 먼저
- 카드형 박스 남발 → 화면 조각남
- 설명 텍스트 과다 → 인터페이스가 아닌 문서처럼
- key color가 배경/카드/텍스트 동시에 → 강조색이 아닌 기본색으로 소비
